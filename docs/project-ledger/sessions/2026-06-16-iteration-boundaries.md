# 2026-06-16 迭代边界与约束

## 本次变化

- 新增 `docs/00-governance/iteration-boundaries.md`。
- 将迭代分为可自动推进、需人工确认、必须人工协助、只进私有边界和暂不承诺五类。
- 明确授权中枢、provider 费用、发布动作、真实平台登录态和知识库镜像的约束。
- 更新 README、用户指引、工作空间治理说明、授权中枢后续规划和待确认问题。

## 核心决策

- 后续默认先做公开安全的本地 dry-run、mock、契约、审计和检查，再进入真实外部系统。
- 发布、真实 provider 调用、Render / Cloudflare 生产配置、GitHub UI 安全设置和真实平台账号操作必须人工确认。
- 微信、朱雀AI、抖音、小红书、知乎、头条等账号登录、验证码、续登和正式发布确认必须由人工协助。
- Cookie、浏览器 Profile、二维码、截图、未脱敏平台原文、真实账号配置和私有 connector 实现不得进入公开仓或知识库镜像。

## 后续建议

- 将 L1 / L2 人工门槛逐步纳入 `next:actions`、`roadmap:next` 或 release decision 机器可读报告。
- 为授权中枢设计统一的人工协助队列，区分续登、验证码、审批、费用批准和发布确认。
- 进入真实平台 connector 前，先准备私有仓实现边界和失败分类测试。
