# 2026-06-16 迭代预期控制

## 背景

用户明确指出，迭代边界的重点不是单纯的权限分级，而是事前明确需求和预期，并在开发过程中发现预期不符时及时处理。目标是防止目标模糊下的过度开发、token 滥用、项目臃肿和需求偏离。

## 本次补充

- 在 `docs/00-governance/iteration-boundaries.md` 中补充“需求和预期边界”的定义。
- 新增事前边界卡：目标、非目标、验收、范围、风险、预算和决策点。
- 新增预期偏差处理机制：暂停扩张、归因、给选项、等确认。
- 新增防臃肿和 token 控制规则：最小可验证增量、复用优先、连续失败暂停、跨系统先拆分。
- 更新 `docs/00-governance/workspace-governance.md`，把预期控制纳入默认流程。

## 后续建议

- 可把“边界卡”做成 issue/PR 模板或 `next:actions` 的机器可读字段。
- 对长任务增加显式 checkpoint：达到范围、时间或 token 阈值后自动汇报偏差和下一步选项。
- 后续如果新增工具，应优先让它输出目标、非目标、验收和人工确认点，而不是直接扩大自动化能力。
