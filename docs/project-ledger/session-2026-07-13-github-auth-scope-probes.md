# GitHub 授权 scope 精确探针

日期：2026-07-13

状态：本地实现、完整回归、新鲜克隆和 GitHub CI 通过；草稿 PR #26 等待审查。仅进行离线响应模拟，不调用真实 GitHub API，不读取真实 token。

## 需求

- 用户目标：让平台授权连接器 P0.2 能分别判断目标仓库的仓库读取、Actions 读取和 Pull Request 读取权限，避免只读仓库元数据就误报全部 scope 可用。
- 成功标准：`repo_read`、`actions_read`、`pull_request_read` 使用各自只读端点；空 workflow run 或空 PR 列表仍返回可用；限流、凭据拒绝、目标不可见、平台故障和网络故障有稳定脱敏结果。
- 输入材料：现有 GitHub 私有适配器生成器、平台任务输入合同和离线测试。
- 输出形态：公开生成器、任务输入验证、自动化测试、用户手册和架构说明。
- 非目标：真实 GitHub 调用、token 配置、写权限、PR 合并、Actions workflow dispatch、通用 GitHub SDK。
- 用户确认点：真实最小权限 token 和非关键仓库验证仍是独立人工门禁。

## 预期开发工作

- 预期产物：scope 到只读 REST 端点的固定映射、完整仓库目标校验、错误分类和离线请求测试。
- 允许改动：`tools/new-github-auth-private-adapter.js`、`src/domain/workflow.js`、对应测试、公开文档与本台账。
- 明确不碰：`runtime/private/`、真实凭据、Auth Hub 远程部署、公众号和小红书实现。
- 实现路径：生成临时私有适配器，替换测试进程中的 `fetch`，验证 URL、方法、响应分类和结果脱敏。
- 工作规模：单一连接器的窄幅行为修正，不新增依赖或服务。
- 新增门槛：若官方端点需要写权限、管理员权限或无法在空仓库稳定判断，立即停止扩围并重新评审。

## 实现结果

- GitHub 任务现在要求同时提供 `owner` 与 `repo`，不再退回 `/user` 后把 token 身份可用误报为仓库 scope 可用。
- `repo_read` 使用只读分支列表，验证 Contents read；`actions_read` 使用 workflow runs 列表，验证 Actions read；`pull_request_read` 使用 PR 列表，验证 Pull requests read。
- 三个端点都使用 `GET` 和 `per_page=1`；返回 200 的空列表仍判定为可用，不读取或保存响应正文。
- 429 或限流头优先判定为可重试限流；401/403/404 归为凭据或目标访问问题；5xx、超时和网络异常归为平台不可用，内部错误文本不出现在结果中。
- 未知 scope 在私有适配器层失败关闭，任务合同层继续拒绝未支持 scope。
- GitHub 官方文档确认三个端点分别要求 Contents、Actions 与 Pull requests 只读权限；公开资源可能匿名可读，因此真实 scope 验收必须使用非关键私有仓。

## 边界控制

- 范围边界：只做 GitHub 只读授权健康检查准确性。
- 成本边界：本轮不产生外部调用和费用。
- 安全边界：测试 token 仅用占位值；不读取本机 `GH_TOKEN` 或 `GITHUB_TOKEN`。
- 权限边界：不申请、不扩大、不验证任何写权限。
- 停止条件：发现需要真实账号、管理员权限、平台写操作或新增大型抽象时停止。
- 偏差处理：无法精确判断的状态返回需要人工处理或平台不可用，不用“ready”掩盖不确定性。
