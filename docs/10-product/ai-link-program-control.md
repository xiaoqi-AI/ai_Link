# AI Link 项目总监控制面

状态：当前执行基线。用于回答项目方向、模块进度、合并顺序和人工决策，不替代单个 PR 的代码审查。

更新时间：2026-07-14

## 使用规则

只在以下场景读取本控制面或运行 `npm run roadmap:next:json`：

- 用户询问项目状态、整体规划、下一步或迭代方向。
- 需要决定 PR 合并、真实平台调用、账号登录、费用、远程部署、DNS、密钥或数据清理。
- 一个任务会跨越 Auth Hub、平台连接器和远程部署两个以上模块。

普通代码修改、UI、文档排版、单元测试和单项目局部修复不读取本控制面。这样可以保持“触发时完整读取，平时零额外上下文”的冷启动模式。

## 总目标

AI Link 当前把三项能力作为一个受治理的项目群推进：

1. 模块 2：Auth Hub 状态中枢，让负责人看到哪些平台、任务或授权需要人工处理。
2. 模块 5：平台授权连接器 P0.2，让 GitHub、公众号和小红书通过统一合同与本机私有执行器接入。
3. 模块 6：Auth Hub 远程化，让其他项目能提交任务、查看脱敏状态和进行审批，但不接收平台 Cookie、二维码或浏览器 Profile。

## 决策卡

### 决策 A：低噪声状态通知是否接线

- 背景：低噪声 watcher 已进入 `main`，但没有通知渠道和持续可用的 Auth Hub 端点。
- 决策内容：是否启用、使用哪个渠道、接收人、频率和停止条件。
- 建议：远程 Auth Hub 验收前只做 repository-local、每日最多一次的检查，且只在 `notify=true` 时通知。
- 价值：只在新增或恶化的授权事项出现时提醒，不让普通项目任务重复查询 AI Link。
- 风险：端点不稳定或频率过高会制造假警报；Access 未验收前连接远程渠道可能暴露运营元数据。

### 决策 B：首个真实平台验收

- 背景：三平台公开合同与私有脚手架已完成，真实账号仍未被公开代码读取。
- 决策内容：批准平台、账号、只读范围、频率、时间窗和停止条件。
- 建议：先 GitHub 只读，再小红书只读，最后公众号健康检查；草稿和正式发布继续分开审批。
- 价值：从低交互、易撤销的路径验证 Auth Hub 真实证据合同。
- 风险：真实调用可能消耗配额、触发风控或暴露账号状态；凭据必须留在本机私有边界。

### 决策 C：Auth Hub 远程部署

- 背景：会产生持续费用、公网 DNS、生产密钥、身份策略和数据库恢复责任。
- 决策内容：创建任何远程资源前，确认域名、区域、规格、邮箱、Service Auth、密钥存放、原生子域名、清理和备份/PITR。
- 建议：`auth.xiao-qi-ai.com`、Singapore、Starter 单实例、`basic-256mb`、精确邮箱、一个可撤销 Service Auth token、首次不开自动 retention、apply 前验证备份/PITR。
- 价值：ParentingGame、Hermes 和其他项目可使用统一远程状态中心，而不持有平台登录态。
- 风险：配置错误可能公开控制台、锁定负责人或让数据清理不可恢复。

## 状态命令

```powershell
npm run roadmap:next:json
npm run next:actions:json
npm run auth-hub:status:json
npm run auth-hub:status:watch:json
npm run auth-hub:remote:next:json
```

- `roadmap:next`：项目方向、三模块进度和决策卡。
- `next:actions`：当前可执行项和人工项。
- `auth-hub:status`：真实平台授权与人工协助状态；只有需要平台状态时运行。
- `auth-hub:status:watch`：低频比较上次脱敏状态，只有新增人工事项或恶化时给自动化 `notify=true`；普通任务不运行。
- `auth-hub:remote:next`：远程部署门禁；只有进入部署决策时运行。

## 更新责任

- PR 合并、真实验收或部署决策发生后，更新本文件、`tools/show-roadmap-next.js` 和相关项目账本。
- 不把会话历史当作项目真源；GitHub PR 状态、代码、测试、部署证据和本文件共同构成当前基线。
- 不记录 token、Cookie、二维码、登录截图、浏览器 Profile、数据库连接串或平台原始响应。
