# 授权中枢后续规划草案

状态：草案。用于承接当前 MVP 后的实施顺序，不代表已经承诺公开能力。

## 当前基线

公开仓当前目标是把“云端私有控制台 + 本地浏览器执行器”的骨架跑通：

- 控制台、任务 API、审批、审计和 mock 连接器已经进入公开 MVP。
- 控制台已能单独展示待人工处理任务，并支持处理后重新排队。
- 连接器能力契约和只读状态 API 已进入公开 MVP，可展示微信、朱雀AI和预留平台的安全状态。
- Provider live 验收通过 Bitwarden Secrets Manager 管理真实 API key。
- Render 部署骨架保留，生产部署前必须经过 Cloudflare Access 和应用内登录双门禁。
- 真实平台登录态、Cookie、二维码、截图和原始平台内容仍只允许放在本机私有目录或私有仓治理材料中。

## 迭代边界

授权中枢后续推进受 `docs/00-governance/iteration-boundaries.md` 约束：

- 可自动推进：mock 流程、连接器契约、状态页、脱敏审计、本地检查和公开文档。
- 需人工确认：Render / Cloudflare 配置、生产环境变量、GitHub Release、npm 发布、真实 provider 调用和费用边界。
- 必须人工协助：微信/朱雀AI等平台登录、扫码、验证码、续登、正式发布确认和真实内容审核。
- 只能进私有边界：Cookie、浏览器 Profile、二维码、登录截图、未脱敏原文、真实账号配置和私有 connector 实现。
- 暂不承诺：真实平台自动发布、云端保存高价值账号登录态、绕过验证码或风控、未经审批的批量调用。

默认推进顺序是先本地 dry-run / mock，再远端空跑；先只读和草稿，再发布能力；先安全扫描和 fresh clone，再 tag / deploy。

## 阶段 1：生产门禁与远端空跑

目标：让 `voice.xiao-qi-ai.com` 可以承载真实任务控制台，但仍只跑 mock 或脱敏任务。

交付：

- Render Web Service + Postgres 按 `render.yaml` 创建。
- Cloudflare Access 保护域名，并配置 Access AUD tag。
- Render 环境变量完成：应用密码、session secret、admin/executor/codex token、数据库和 Access origin guard。
- 本地执行器通过 Cloudflare Service Auth 连接远端。
- 远端 smoke 可创建任务、执行 mock 链路并写回结果。

验收：

- `npm run auth-hub:deploy:check` 在生产变量注入后通过。
- `npm run auth-hub:remote:smoke` 可穿过 Access 完成测试任务。
- 未授权浏览器不能进入控制台；授权后仍需要应用内登录。

## 阶段 2：真实微信和朱雀AI私有连接器

目标：把真实取材、检测和草稿能力放进私有边界内验证。

交付：

- 私有仓或本机 `runtime/private/` 保存连接器配置和浏览器 profile。
- 微信取材优先走官方接口；页面登录态只作为接口不足时的补位。
- 朱雀AI优先网页连接器；若发现稳定官方 API，再切换 API 连接器。
- 失败原因分类：登录过期、验证码、限流、页面变化、检测服务失败。

验收：

- 单条测试链接可取材、检测、生成草稿摘要。
- 任何正式发布仍必须经过控制台或邮件审批。
- 日志和云端结果只保留脱敏摘要，不上传私密截图或原始账号状态。

## 阶段 3：内容工作流产品化

目标：把“链接取材到公众号草稿”的常用路径做成稳定操作台。

交付：

- 任务模板：链接取材、热点整理、朱雀检测、改写、公众号草稿、指标回收。
- 控制台展示任务队列、等待人工处理事项和审批历史。
- 邮件只发提醒和控制台链接，不放正文、截图、token 或平台内容。
- 发布前审批固定为必选，不做默认自动发布。

验收：

- 任务状态能清楚区分排队、执行中、待人工、审批中、完成和失败。
- 每次失败能给出下一步操作建议，例如续登、重试、稍后执行或人工处理。

## 阶段 4：多平台预留到真实接入

目标：逐步接入抖音、小红书、知乎、今日头条等平台，但每个平台先通过能力契约。

统一能力：

- `read_content`
- `detect`
- `create_draft`
- `publish`
- `metrics`

推进顺序建议：

- 先做只读能力和草稿能力，再评估发布能力。
- 先接低风险平台或官方 API 明确的平台。
- 每个平台上线前必须补充连接器契约测试、失败场景测试和敏感数据扫描。

## 阶段 5：治理和运维增强

目标：让系统适合长期使用，而不是只能靠临时人工看护。

交付：

- 审批超时策略、续登提醒、执行器离线提醒。
- 任务和 artifact 保留策略自动清理。
- 私有仓和公开仓的脱敏发布流程。
- 知识库镜像只同步公开说明、架构决策和脱敏交接。
- 生产监控：健康检查、任务失败率、队列长度和邮件发送失败。

## 需要确认的问题

- 第一批真实连接器优先顺序：微信取材、朱雀AI检测、公众号草稿、指标回收。
- 本地执行器是否只跑在当前电脑，还是准备一台常在线 Windows 机器。
- 邮件审批入口使用哪个收件邮箱，以及是否需要移动端审批体验。
