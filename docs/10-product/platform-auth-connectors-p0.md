# 平台授权与采集连接器 P0 计划

状态：P0.1 已完成；P0.2 已完成交互登录审批门禁；P0.3 已具备公众号健康检查私有适配器脚手架，真实凭据与调用仍受人工验收约束

需求来源：Hermes Agent 脱敏交接，2026-07-11

负责人：AI Link 项目

## 项目结论

AI Link 接受“平台授权与采集连接器”需求，但不会把它实现成另一套通用抓取器。AI Link 负责本机登录态、人工协助状态、连接器执行和脱敏结果；Hermes 继续负责热点业务、来源真实性、三平台覆盖和内容生产。

第一轮只完成公开仓 P0.1：

- 统一会话状态和结果合同。
- `platform_auth_collect` 最小任务入口。
- 本机 `runtime/private/` 私有连接器注入机制。
- `ready`、`needs_action`、`blocked` 的稳定映射。
- URL、日志、任务结果和连接器状态的脱敏测试。

真实小红书登录、公众号凭据、出口 IP 白名单和草稿创建不属于 P0.1。它们分别进入 P0.2、P0.3 的私有实现与人工验收。

## 背景与价值

当前问题不是缺少更多模型，而是平台登录和采集状态没有统一归属：

- 小红书需要本机扫码、验证码、续登和真实搜索验证。
- 公众号官方 API 需要区分凭据错误、出口 IP 白名单、限流和服务不可用。
- 上游业务不能把“有 Cookie 文件”或“连接器没有报错”当成真实采集成功。

AI Link 建立统一合同后，上游只消费有限的公开状态和具体内容链接，不再直接管理 Cookie、浏览器 Profile 或二维码生命周期。主要价值是减少重复登录集成、关闭伪成功，并让人工协助有明确入口和可重试断点。

## 责任边界

| 模块 | 负责 | 不负责 |
| --- | --- | --- |
| AI Link Auth Hub | 任务状态、人工协助、重试、脱敏审计 | 内容排名、真实性终审、文章生成 |
| AI Link 本机执行器 | 加载受信任的私有连接器，执行只读采集或健康检查 | 云端保存登录态、绕过验证码 |
| Hermes | 消费脱敏结果，重新校验平台、域名、具体链接和覆盖门禁 | 直接管理平台 Cookie 和扫码生命周期 |
| 私有连接器 | 与具体平台交互并返回公开合同 | 把原始响应、账号信息或登录态交给 Auth Hub |

## 已确认决策

| 决策 | 结论 | 价值 | 主要风险与控制 |
| --- | --- | --- | --- |
| 私有实现位置 | 代码进入 `ai_Link-internal`，运行态只在本机 `runtime/private/` | 公开仓可审计，登录态不出本机 | 私有模块拥有本机代码执行权，只加载明确配置的受信任文件 |
| 首批执行器 | 只运行在当前 Windows 电脑 | 最短路径复用现有登录环境 | 电脑离线时任务不可执行，后续再评估常在线执行器 |
| Auth Hub 模式 | local-first；远端只保存任务状态和脱敏结果 | 降低高价值登录态上云风险 | 远端状态可能晚于本机，以执行器回写为准 |
| 小红书范围 | `checkSession`、`beginLogin`、`readContent`；只读 | 满足热点搜索，不引入互动风险 | 扫码、验证码、风控必须由用户完成 |
| 公众号范围 | 官方 API 优先；首批健康检查，随后只到草稿 | 错误可诊断，优先使用官方能力 | 固定出口 IP 和白名单是外部前提 |
| 正式发布 | 每次人工审批，P0 不实现自动正式发布 | 防止误发和账号事故 | 审批不可被重试或定时任务绕过 |
| IP 白名单 | 固定出口 IP + 公众号后台白名单 | 解决真实根因 | 不把白名单错误误判为重新扫码 |

这些决策批准的是架构和公开合同，不等于批准任何真实登录、真实凭据读取、外部写操作或正式发布。

## 分阶段实施

### P0.1：公开合同与私有加载机制

完成条件：

- `GET /api/connectors` 能描述必备能力、可选能力、模式和稳定问题代码。
- 私有模块只能从仓库本机 `runtime/private/` 加载。
- 私有连接器缺少必备方法时返回 `connector_contract_failed`。
- `platform_auth_collect` 只接受批准的平台和操作。
- `begin_login` 先返回 `approval_required/interactive_approval_required`，不会直接调用私有连接器；用户批准后才进入本机交互登录步骤。
- 小红书结果只允许 1 至 4 条已验证具体笔记链接，并移除全部查询参数和 fragment。
- `session` 只公开 `state` 与 `checked_at`。
- 原始 HTML、原始响应、Cookie、token、二维码、localStorage 和 Profile 不进入任务结果或日志。

### P0.2：小红书真实只读

人工协助点：

1. AI Link 返回 `needs_action/login_required` 或 `login_expired`。
2. AI Link 对 `begin_login` 创建 `platform_interactive_login` 审批；未批准时不会打开本机浏览器或调用私有连接器。
3. P0.2 交互门禁获批后，由用户在本机可见浏览器完成扫码、验证码或续登。
4. AI Link 重试同一任务并执行真实搜索。
5. 只有取得具体笔记 URL 且可达性为 `verified` 才返回 `ready`。

停止条件：出现平台风控、反复验证码、账号异常、需要互动/发布或平台条款不明确时停止自动推进。

### P0.3：公众号官方 API

公开脚手架已完成：维护者可以在不提交凭据的前提下生成 `runtime/private/wechat-official-health-adapter.mjs`。它只把 `check_health` 替换为私有官方 API 健康检查，其余公众号能力继续使用 mock；真实调用尚未验收。

人工协助点：

1. 凭据由用户写入本机 secret manager 或受控环境，不进入聊天和 Git。
2. AI Link 运行健康检查并区分凭据、白名单、限流和不可用。
3. 若为 `official_api_ip_not_whitelisted`，用户配置固定出口 IP 或公众号后台白名单。
4. 健康检查通过后再单独验收草稿创建；真实标识只留在私有运行态。

正式发布仍是独立需求和每次人工审批项。

### P0.4：Hermes 联调

Hermes 创建 `platform_auth_collect` 任务，AI Link 返回脱敏结果。Hermes 必须再次验证：

- 平台枚举与 URL 域名一致。
- URL 是具体内容页。
- `source_reachability.status` 为 `verified`。
- `acquisition_provider` 如实标记。
- 平台覆盖门禁不因单个 `ready` 字符串而自动放行。

## P0.1 迭代边界卡

### 需求

用户目标：让 AI Link 安全接收 Hermes 的平台授权连接器需求，并形成可供私有实现复用的公开合同。

成功标准：私有连接器可被本机执行器加载，错误不会伪成功，公开结果不含运行态或账号数据。

输入材料：Hermes 脱敏交接、现有 Auth Hub、connector registry、本地执行器与脱敏模块。

输出形态：代码、测试、中文文档、项目账本和独立 PR。

非目标：真实扫码、公众号密钥、固定出口设置、草稿写入、正式发布、今日头条迁移。

用户确认点：P0.2 真实登录、P0.3 凭据与白名单、任何真实写操作。

### 预期开发工作

允许改动：`src/connectors/`、`src/domain/`、`src/executor/`、必要的存储/脱敏代码、对应测试和文档。

明确不碰：`runtime/private/` 中的既有文件、Hermes 代码、Coze 主流程、真实账号和远端部署。

实现路径：复用现有 registry、`runTask`、`needs_action`、retry 和 `redact`，不新增依赖或第二套队列。

### 验证

- 合同和任务流定向测试。
- 全量 `npm test` 与类型检查。
- 安全扫描、治理检查和 fresh-clone 验证。
- Git diff 检查、公开仓同步和知识库镜像验证。

### 边界控制

- 私有模块路径不接受任务输入，只接受本机环境配置。
- 任何原始平台响应先经过私有连接器，再由公开 allowlist 重建结果。
- 一旦需要真实登录、凭据、外部写操作或平台规避手段，立即停止并进入人工门禁。

## 当前人工决策建议

P0.2 的公开交互门禁已就绪。下一步若要真实测试小红书，只需要用户单独确认测试账号、允许的只读关键词、测试时间窗和停止条件；P0.3 开始前应确认公众号账号权限、固定出口方案、调用配额和草稿验收内容。
