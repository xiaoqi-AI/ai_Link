# 公开 GitHub 维护规则

## 维护目标

这个仓库是公开仓库。公开仓库的维护重点不是只把代码推上去，而是让陌生访问者能判断项目是什么、当前能做什么、如何反馈问题，以及哪些内容不能公开。

项目同时使用私有内部 companion 仓 `xiaoqi-AI/ai_Link-internal`。公开仓只承载可公开内容；内部仓用于未公开路线、实验、供应商评估、运维说明和公开发布门禁。

## 每次更新必查

- `README.md` 是否仍然准确。
- `docs/user-guide.md` 是否覆盖新的使用方式、限制或入口。
- `.github/ISSUE_TEMPLATE/` 是否能引导用户提交有效反馈。
- `CONTRIBUTING.md` 是否能说明协作预期。
- `SECURITY.md` 是否说明安全问题不要直接公开细节。
- `npm run release:readiness` 是否通过，且人工确认项是否有明确 owner 或后续动作。
- 是否有敏感信息、账号信息、二维码、登录态、私密截图或个人数据误入 Git。
- 本地 `main` 是否已推送并与 `origin/main` 对齐。
- 是否有来自内部仓的内容需要先脱敏、改写和补充用户指引。

## 建议的 GitHub 设置

- 仓库可见性：public。
- 默认分支：`main`。
- 主分支保护：建议禁止删除和强推；成熟后可要求 PR、状态检查或 review。
- Issues：开启，用模板收集问题和需求。
- Discussions：如果以后有外部用户交流需求，再开启。
- License：Apache-2.0。

## 提交习惯

- 提交信息短而具体，例如 `docs: add public repo maintenance guide`。
- 一次提交只解决一个清晰主题。
- 文档和用户指引改动要和功能改动一起提交，避免功能已经变化但说明滞后。
- 推送后检查 GitHub 页面是否能从 README 顺利找到用户指引、贡献说明和安全反馈入口。
- 从内部仓同步到公开仓时，提交信息和文档只描述公开后的用户价值，不暴露内部来源细节。

## 收尾口径

每次汇报至少说明：

- 改动内容
- 验证结果
- GitHub 是否已推送
- 知识库镜像是否已同步
- 用户指引是否需要继续补充
