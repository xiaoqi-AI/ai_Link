# 迭代边界与约束

状态：有效治理约定。用于判断后续 AI Link、授权中枢、provider、connector 和发布流程迭代时，哪些可以自动推进，哪些必须等待人工确认。

## 目标

本文件把每次迭代前后的需求、预期、验证和边界写成可复用机制，避免在目标模糊时继续开发，导致 token 滥用、功能扩张、项目臃肿，甚至偏离用户真实需求。

迭代边界首先是需求和预期边界，其次才是权限、安全和发布边界。它要求在开发前说清楚“本轮到底要完成什么”，在开发中持续比对“实际推进是否仍符合预期”，一旦偏离就暂停扩张、复盘原因、给出选项，必要时等待用户确认。

核心原则：

- 每次迭代只追一个主目标；相关但非必要的想法进入待确认问题、项目账本或下一轮。
- 需求、预期开发工作、验证方式和边界控制四项不清楚时，不进入大规模实现。
- 先复用现有文档、脚本、provider、connector、route、store 和 UI 模式，再考虑新增能力。
- 先做本地 dry-run、mock、只读验证和脱敏审计，再进入真实 provider、真实平台和发布动作。
- 任何会产生真实外部影响、费用、账号风险、公开承诺或敏感数据处理的动作，都必须有人确认。
- 公开仓只承载可公开代码、文档、mock、脱敏报告和治理流程；私有仓、本机 `runtime/private/` 和外部 secret manager 承载真实登录态、密钥、二维码、截图和未脱敏平台内容。

## 迭代边界卡

每次进入实质开发前，默认先形成一张轻量边界卡。简单任务可以在计划里简写，复杂任务应写入 issue、项目账本、设计文档或交接说明。

边界卡必须覆盖四块：需求、预期开发工作、验证、边界控制。它的作用是让 Codex、目标模型和其他 Agent 在开发前对目标、工作量、验收方式和停止条件达成一致，避免在预期不清时持续消耗 token，或者把过渡性实现扩张成长期平台负担。

### 需求

需求用于回答“为什么做、做成什么才算对”。至少明确：

- 用户目标：本轮结束后，用户能看到、使用、决策或交接什么。
- 触发背景：这次需求来自用户明确指令、前序阻塞、质量问题、治理要求，还是探索候选。
- 成功标准：用一句话描述完成后的可观察结果。
- 输入材料：本轮依据哪些用户要求、文档、代码、外部前提或人工确认。
- 输出形态：最终应交付代码、文档、脚本、报告、配置、镜像同步、操作清单，还是只给方案。
- 非目标：本轮明确不做什么，哪些只是记录、研究或后续候选。
- 优先级：本轮最重要的一个结果是什么；如果只能完成一件事，先保哪一件。
- 用户确认点：哪些判断不能由 Codex 代替用户决定，例如发布、真实费用、账号权限、产品方向或外部平台动作。

如果无法说清用户目标、成功标准或非目标，应先补需求，不应直接扩代码、补工具或做平台化设计。

### 预期开发工作

预期开发工作用于回答“为了满足需求，合理的工作量和改动范围是什么”。至少明确：

- 预期产物：代码、文档、脚本、配置、测试、报告、镜像同步或操作清单。
- 触达范围：预计会改哪些目录、模块和文档；哪些目录明确不碰。
- 实现路径：优先沿用的现有模式、命令、工具和抽象。
- 工作规模：预计是小修、单文件文档、局部功能、跨模块改动，还是需要拆阶段。
- 影响面：会影响用户入口、CLI、provider、connector、授权中枢、发布流程、知识库镜像或 CI。
- 新增门槛：新增脚本、依赖、抽象、页面、长期流程前，必须说明它解决的真实重复、风险或维护成本。
- 资源预算：限定本轮探索深度、工具调用规模和验证成本；超过预算时停止复盘。

如果一个小需求开始需要多个新脚本、多层抽象、跨三个以上子系统或新平台能力，默认视为范围扩大，应拆成下一轮或重新确认。

### 验证

验证用于回答“怎么证明没有偏离需求，而且没有破坏安全边界”。至少明确：

- 功能验证：用户目标对应的命令、测试、页面检查、dry-run、smoke 或人工验收项。
- 回归验证：对受影响模块运行现有检查，例如 `npm.cmd run check`、`npm.cmd run security:scan`、`npm.cmd run release:readiness:json` 或项目专用脚本。
- 安全验证：确认没有 `.env`、token、API key、二维码、登录态、截图、未脱敏内容或 `runtime/private/` 数据进入 Git、知识库、issue、PR 或聊天。
- 状态验证：确认 Git 工作区、远端同步、知识库镜像和 CI 状态是否符合本轮完成定义。
- 证据边界：验收证据只记录公开安全摘要、命令状态、文件路径和决策说明，不记录真实凭据、原始 provider 响应、平台后台截图或私有内容。

如果验证无法证明用户目标已经达成，应交接为“未完成/需要确认”，不能用通过了无关检查来替代需求验收。

### 边界控制

边界控制用于回答“什么时候继续，什么时候停”。至少明确：

- 范围边界：只处理本轮需求直接需要的文件、模块和文档。
- 成本边界：控制 token、时间、工具调用、网络调用和真实 provider 费用。
- 安全边界：不读取、不保存、不打印、不提交真实密钥、登录态、二维码、截图和私有数据。
- 权限边界：GitHub UI、Bitwarden、Render、Cloudflare、npm publish、真实平台登录和发布动作属于人工确认或人工协助。
- 产品边界：未确认的方向只写为草案、待确认问题或候选路线，不直接升级成承诺能力。
- 停止条件：目标不清、验证失败多轮、范围扩大、触碰 L1 / L2 / L3、发现敏感内容、外部前提缺失或成本超界时，停止扩张并复盘。

边界控制不是让迭代变慢，而是让每一轮只做有验收价值的最小可交付增量。

## 预期偏差处理

开发过程中，实际情况和边界卡不一致时，应视为预期偏差。

常见偏差包括：

- 用户目标仍没被满足，但实现开始转向工具化、平台化或附带重构。
- 原本是文档或小修，实际变成跨模块改动、新依赖、新服务或新流程。
- 为了通过验证而不断新增补丁，但失败原因其实是需求、架构或外部前提不清楚。
- 实现触碰了未声明的目录、平台、发布流程、真实账号、真实费用或私有数据。
- 输出开始包含未确认承诺、真实平台细节、敏感信息或无法公开的证据。
- token、时间或工具调用明显超过本轮目标价值。
- 产出变成“顺手完善很多东西”，但这些完善没有被用户明确要求。
- 目标模型或外部 Agent 给出了看似可行但超出本轮需求的过渡方案，需要新增长期配置、状态、适配层或自动化来承接。

出现预期偏差时，默认执行五步：

1. 暂停扩张：停止新增功能、脚本、抽象、页面和长期流程，只保留必要诊断。
2. 对照边界卡：逐项比较需求、预期开发工作、验证和边界控制，指出偏离点。
3. 归因：说明偏差来自需求不清、外部前提缺失、技术路径错误、权限不足、验证失败、范围膨胀，还是成本超界。
4. 给选项：给出缩小范围、按当前方向继续、拆成下一轮、转私有边界、等待人工确认或放弃本轮尝试等候选。
5. 等确认或最小修正：涉及 L1 / L2 / L3、产品承诺、真实费用、发布动作或明显范围扩大时，必须等用户确认；普通 L0 小偏差可以选择最小修正并在交接中说明。

严禁在预期偏差已经出现后，用继续写代码来掩盖需求不确定性。

偏差处理优先级：

- 能用文档、测试或现有配置收敛的，不新增平台能力。
- 能留到下一轮的，不在本轮提前实现。
- 需要真实账号、真实费用、发布承诺或敏感数据的，必须转人工门禁。
- 需要跨三个以上子系统的，先拆分成独立迭代并重新确认边界。

## 防过度开发和 token 控制

默认采用“最小可验证增量”：

- 一个迭代只解决一个主问题；相关想法先记录，不抢占本轮目标。
- 一个改动最好能用一句话说明用户价值；说不清时先重新定义目标。
- 长时间探索前先限定问题和搜索范围，不做全仓泛读式排查。
- 同一问题连续失败两轮，应暂停并写明阻塞点，不继续盲试。
- 如果实现需要跨越三个以上子系统，应先拆分阶段并确认优先级。
- 如果只是为了“以后可能有用”，默认不做，先记录为候选。
- 如果目标模型输出开始要求建立大而全的中台、统一协议或长期框架，应回到本轮需求判断是否真的需要。
- 前端和控制台迭代优先补现有页面和状态，不默认新增完整新应用。

新增复杂度的门槛：

- 新增抽象前，必须能说明它减少了真实重复、降低了风险或匹配已有模式。
- 新增脚本前，必须确认没有已有脚本可以扩展。
- 新增文档前，必须确认它不是已有文档的一小节即可承载。
- 新增依赖前，必须确认标准库或已有依赖无法合理完成。
- 新增远端流程前，必须先有本地 dry-run、只读检查和人工确认点。

默认优先级是：文档约束和小范围实现 > 检查脚本 > 机器可读报告 > 平台化能力 > 真实外部自动化。

## 迭代分级

### L0：可自动推进

Codex 可以直接推进并提交的范围：

- 文档补充、公开说明、项目账本和脱敏架构记录。
- mock connector、dry-run provider、测试夹具和本地只读检查。
- 安全扫描、包内容检查、release readiness、fresh clone 和治理脚本。
- 控制台展示脱敏摘要、状态、审计 metadata 和公开安全报告。
- 新增不会触发外部调用、不会读取密钥、不会发布内容的本地工具。

约束：

- 必须保持公开仓不含 `.env`、token、Cookie、二维码、浏览器 profile、截图、真实平台原文和私有路径。
- 影响用户入口、使用方式或安全边界时，要同步更新 `README.md`、`docs/user-guide.md` 或 `docs/00-governance/`。
- 重要迭代结束前要运行对应验证，并同步 GitHub 与知识库镜像。

### L1：需要人工确认后推进

Codex 可以准备方案、脚本、检查清单和预览，但执行前需要人工确认：

- Git tag、GitHub Release、npm publish 或任何公开发布动作。
- GitHub branch protection、required checks、secret scanning、push protection 等 UI 设置。
- Render、Cloudflare Access、SMTP、域名、数据库等生产环境配置。
- Bitwarden Secrets Manager 项目、machine account、`BWS_ACCESS_TOKEN`、GitHub Environment Secret。
- 真实 provider 调用、provider-live workflow dispatch、模型费用边界和外部 API 配额使用。
- `release:decisions:update -- --yes` 这类会把人工决策写入公开记录的动作。

约束：

- 人工确认应留下公开安全证据，例如“已在 GitHub UI 开启 branch protection”，不要写入 token、账号细节或截图。
- 预览命令和检查报告可以进公开仓；真实凭据、一次性 token 和平台后台截图不进公开仓。

#### 已批准迭代的 PR 合并执行

维护者已经批准本轮需求、迭代范围或明确授权后续合并时，PR 的机械执行可按 L0 自动推进，不再逐个请求相同确认：

- Codex 可以调整堆叠 PR 的基线、重放提交、解决不改变已批准行为的冲突、等待全新 CI，并按依赖顺序合并。
- 每个 PR 必须使用普通合并流程，required checks 必须在当前头提交上通过；不得使用管理员绕过、关闭保护或合并失败/等待中的检查。
- 冲突解决必须保留已经进入 `main` 的安全边界和验收，不得借机扩大产品范围、增加真实外部调用或改变人工门禁。
- 若冲突涉及产品取舍、架构方向、生产配置、真实账号、密钥、费用、公开发布或不可逆外部动作，仍按 L1/L2 暂停并向维护者完整汇报。
- 合并完成后必须确认本地 `main`、`origin/main` 与 GitHub 远端一致，并执行对应测试、安全扫描、治理检查和知识库镜像校验。

### L2：必须由人工协助

这些动作无法由 Codex 单独完成，必须由你或授权维护者参与：

- 微信、朱雀AI、抖音、小红书、知乎、头条等账号登录、扫码、验证码、短信、二次验证和续登。
- 选择 Cloudflare Access 授权邮箱范围、审批收件邮箱、生产域名和外部服务套餐。
- 确认真实内容是否可发布、是否需要修改、是否符合账号定位和平台规则。
- 确认真实 provider 调用预算、模型选择、费用上限和是否允许 live 验收。
- 确认 npm 发布渠道、GitHub Release 公开时机和回滚策略。

约束：

- Codex 可以提示“需要续登”“需要验证码”“需要审批”，但不能绕过平台认证或替你确认高风险动作。
- 正式发布内容必须每次人工确认；系统只能生成草稿、摘要和审批入口。

### L3：只进私有边界

以下内容只能进入私有仓、本机私有目录或外部 secret manager：

- 平台 Cookie、localStorage、session、浏览器 profile、二维码、登录截图。
- 未脱敏平台原文、私密网页 HTML、账号后台数据、联系人或用户信息。
- 真实 API key、BWS token、Cloudflare service token、SMTP 密码、数据库连接串。
- 内部供应商评估、未公开运营策略、真实账号配置和私有 connector 实现细节。

约束：

- 公开仓只保留合同、mock、脱敏接口、错误分类和公开安全说明。
- 知识库镜像只同步公开说明、架构决策和脱敏交接；不接收运行态和登录态。

### L4：暂不承诺

在没有单独确认前，以下能力不作为当前 MVP 承诺：

- 真实平台自动发布。
- 云端保存高价值账号浏览器登录态。
- 自动绕过验证码、风控、人机验证或平台频控。
- 对平台条款不明确的抓取、批量搬运、批量私信或自动互动。
- 未经成本审批的真实模型批量调用。
- 未经发布决策的 npm 包发布或正式公开版本承诺。

## 推进顺序约束

后续迭代默认按下面顺序推进：

1. 先文档和契约，再实现。
2. 先本地 mock / dry-run，再真实外部调用。
3. 先只读和草稿，再发布能力。
4. 先脱敏审计，再保存或同步结果。
5. 先安全扫描和 fresh clone，再 tag / release / deploy。
6. 先人工门槛关闭，再进入生产或真实账号。

任何阶段如果发现敏感内容、登录失效、验证码、平台限流、费用超界、Git 冲突或发布决策未确认，应停止自动推进并明确列出需要人工处理的事项。

## 授权中枢专项约束

- Render 端只保存任务、审批、审计和脱敏 artifact。
- 本地执行器可以使用本机浏览器 profile，但 profile 必须放在 `runtime/private/`，不上传 Render、不进 Git、不进知识库。
- `GET /api/connectors` 只展示平台、状态、能力和问题代码，不展示私有实现或登录态。
- `POST /api/tasks/:id/approve` 只能用于确认继续或确认发布；正式发布仍必须每次人工确认。
- 邮件只发提醒和控制台链接，不放正文、截图、token 或平台内容。
- 真实 connector 上线前必须补契约测试、失败分类、脱敏检查和人工处理路径。
- 状态页、定时刷新和普通任务不得自动触发真实 connector 探测。只有明确设置 `evidenceIntent=connector_probe` 的 allowlist 只读健康任务可以生成证据，且真实执行属于 L1 外部调用门禁。
- probe 必须绑定 token 对应 executor、进程 session、当前一次性 lease 和新鲜 private heartbeat；mock、重放、旧租约或过期证据一律失败关闭。
- 所有执行器结果都必须绑定领取任务的 executor、进程 session 和当前一次性 lease；未领取、错误绑定、过期租约、终态改写或重放一律失败关闭，不能只对 probe 执行该约束。
- GitHub probe 证据必须按批准 scope 与目标仓库隔离；目标只能保存服务端 HMAC 摘要，公开状态不得暴露 owner/repo 或内部摘要，也不得跨目标或 scope 复用成功证据。
- `verifiedOperations` 只证明列出的健康操作；不得据此绕过登录/发布审批，或推导整个平台、写权限和正式发布能力可用。
- 公网 Auth Hub 一旦启用 Cloudflare Access origin guard，就必须校验应用 JWT 的签名、issuer、audience 和身份声明；配置缺失或邮件身份不一致不得退化为信任转发请求头。
- Cloudflare 用户 JWT 与 service-token JWT 必须分开识别；`common_name` 不能作为用户邮箱，service token 必须显式允许。
- 携带 Bearer 或 Cloudflare Service Auth 的 Auth Hub 客户端只能访问显式批准的 HTTPS hostname，必须关闭自动重定向；loopback 不得附加 Service Auth，建议域名也不能作为隐式白名单。
- 应用内控制台会话必须包含服务端校验的绝对到期时间；浏览器 `Max-Age` 不能作为唯一过期依据，旧格式或畸形 Cookie 一律失败关闭。

## Provider 和费用约束

- dry-run 默认安全，可自动执行。
- 真实 provider 调用必须满足 policy gate，并需要显式审批或通过已确认的发布/验收流程。
- provider-live 报告只保存安全摘要，不保存原始 prompt、原始 output 或密钥。
- 成本上限、模型范围和 live 验收时间窗口必须由人工确认。

## 发布约束

- `release:plan`、`release:readiness`、`release:evidence` 和 `package:check` 都是检查，不是发布。
- 创建 `v0.1.0` tag、GitHub Release 或 npm publish 前，必须关闭 manual gates 或明确记录 waiver。
- npm 发布必须先做 `npm publish --dry-run --access public`，并确认账号、包名、版本、包内容和回滚策略。
- 发布后如果发现敏感内容或错误包面，优先撤回/弃用发布并更新安全说明，不做静默覆盖。

## 完成定义

一次有效迭代完成时，应至少说明：

- 改了什么。
- 做过哪些验证。
- GitHub 是否已同步。
- 知识库镜像是否已同步。
- 哪些事项仍需要人工确认。
- 是否触碰了真实账号、真实费用、发布动作或私有数据。

如果触碰了 L1/L2/L3 边界，还应补充证据位置和风险说明；如果只做 L0，本地检查和公开文档记录即可。
